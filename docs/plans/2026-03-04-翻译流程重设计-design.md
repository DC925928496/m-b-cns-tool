# 翻译流程重设计（WPF + CLI + DLL 动态注入）Design

## 目标

按《骑马与砍杀2：霸主》（Bannerlord）本地化规范，重设计工具的完整翻译流程，满足以下体验：

1. 选择需要翻译的 Mod 根目录（允许传父目录，自动定位 `SubModule.xml` 所在模块根）。
2. 勾选是否扫描解析 DLL（**仅限** `TextObject` 构造函数首参为字符串字面量的场景）。
3. 按规则扫描 Mod 文本 XML/JSON 与 DLL，生成统一的“翻译条目列表”，以 **所属文件 / id / text / 译文** 展示并可编辑。
4. 点击“开始翻译”，使用谷歌翻译自动填充空译文，并严格保留占位符。
5. 人工校对并保存记录。
6. 点击“打包”，生成**依赖原 Mod 的外挂汉化 Mod**（外置包，不修改原 Mod），必要时包含运行时注入 DLL。
7. 保存翻译记录；后续再次选择同一 Mod 时，翻译优先复用已保存记录。

本次设计默认破坏性改动：不保证兼容旧的 CLI 参数、旧的 review 文件结构与旧的缓存键策略。

## 规范依据（交叉验证）

- 语言文件目录结构、语言短码（含 `CNs`/`CNt`）以及 `language_data.xml` 的 `LanguageFile xml_path=...` 机制：https://docs.bannerlordmodding.lt/modding/localization/
- `TextObject` 的 `{=stringID}默认文本` 约定（字符串 ID）以及“可本地化文本应放入 `std_module_strings_xml.xml`”的规范：https://docs.bannerlordmodding.com/_csharp-api/localization/ 与 https://docs.bannerlordmodding.com/_csharp-api/localization/textobject
- 占位符 `{...}` 需要保留（示例：`{WORDS}` 由游戏/Mod 运行时替换）：https://www.nexusmods.com/mountandblade2bannerlord/articles/193
- 社区存在“硬编码本地化字符串提取”工具（说明硬编码字符串是常见问题）：https://github.com/BUTR/Bannerlord.LocalizationParser

## 非目标（明确不做）

- 不对原 Mod 的 DLL 做二进制重写（不 patch 原文件）。
- 不扫描/翻译所有 `ldstr` 字面量；DLL 扫描严格限定为 `TextObject` 构造调用。
- 不尝试“运行时注册新的本地化 XML/条目”的不确定 API（以注入替换为主，必要时落盘到语言文件）。
- 不新增额外安全机制（只保持最低安全基线：失败隔离、版本门禁、占位符安全校验）。

## 术语与约定

- **模块根目录**：包含 `SubModule.xml` 的目录。
- **语言文件**：`ModuleData/Languages/**.xml` 下包含 `<string id="..." text="...">` 的 XML。
- **本地化 ID 形式**：`{=id}默认文本`。规则：
  - “字符串以 `{=id}` 开头”视为本地化 ID 前缀；
  - “字符串内部出现 `{=xxx}`”视为普通占位符，必须保留，不当作前缀处理。
- **硬编码文本**：DLL 中 `new TextObject("...")` 的字符串字面量入参且不以 `{=id}` 开头。
- **严格安全优先**：无法可靠检测游戏版本或不在允许范围时，运行时注入功能完全禁用。

## 总体流程（WPF / CLI 共用）

### 阶段 1：扫描（Scan）

输入：
- `modPath`：Mod 路径（模块根或其父目录）
- `scanDll`：是否扫描 DLL（仅 TextObject 字面量）
- `targetLanguage`：默认 `zh-CN`（影响语言文件优先目录：`CNs`/`CNt`）

输出：
- **翻译工程记录文件**（Project Record，JSON）
- UI/CLI 展示的翻译条目列表（来自记录文件）

扫描行为：
1. 定位模块根目录（找到唯一 `SubModule.xml`）。
2. 扫描 `ModuleData/Languages/` 下所有 `*.xml`，构建“已存在 string id 集合”（用于 `{=id}` 去重）：
   - 若存在 `ModuleData/Languages/language_data.xml`，优先读取其定义并优先检查目标语言文件夹（`CNs`/`CNt`）。
   - 若缺失/异常，则遍历 `ModuleData/Languages/**/*.xml` 全部收集。
3. 扫描 Mod 的非语言 XML/JSON（排除 `bin/`、`ModuleData/Languages/` 等）：
   - 仅收集出现 `{=id}` 的引用（用于“语言条目补全候选”），不直接把这些文件的文本作为可翻译目标。
4. 若勾选 `scanDll`：
   - 仅扫描 `SubModule.xml` 声明的 DLL（白名单）。
   - 在 IL 中匹配 `TaleWorlds.Localization.TextObject::.ctor(string, ...)` 的 `newobj` 指令序列，提取首参为字符串字面量的值。
   - 对字面量分类：
     - **ID 前缀类**：以 `{=id}` 开头：
       - 若 `id` 已存在于语言文件集合：不进入翻译列表；
       - 否则：进入翻译列表，并在打包阶段落盘到语言文件（作为 `<string id="id" text="译文">`）。
     - **硬编码类**：不以 `{=id}` 开头：
       - 进入翻译列表，打包阶段写入运行时注入映射（`runtime_localization.json`）。

### 阶段 2：自动翻译（Translate）

输入：
- 工程记录文件
- 翻译引擎：谷歌翻译（默认）/自定义 OpenAI 兼容接口（可选）

行为：
- 仅对“译文为空”的条目调用翻译（保留人工校对内容）。
- 对每条翻译结果做**占位符安全校验**：
  - 必须保持 `{...}`、`{{...}}`、`<...>`、`%...%` 等 token 数量与内容一致；
  - 不安全则回退为源文（不写入译文）。

输出：
- 更新后的工程记录文件（含译文）
- 同步写入 SQLite 缓存（便于跨 Mod 复用）

### 阶段 3：校对（Review）

WPF：
- DataGrid 展示并允许直接编辑“译文”列；
- 支持筛选：仅未翻译 / 仅 DLL 硬编码 / 仅 DLL 缺失 id / 按所属文件过滤。

CLI：
- 提供 `--review <file>` 输出/读取记录文件路径，交由用户用编辑器修改。

### 阶段 4：打包（Package）

输出：
生成外挂汉化 Mod（外置包），满足：
- 模块名：`{原模块名}_CNs`（或按目标语言变体）
- `SubModule.xml`：
  - `DependedModules` 依赖原 Mod（保持“外挂依赖”）
  - `SubModules`：
    - 默认为空（不加载原 Mod 运行时代码）
    - **当存在硬编码注入条目且译文有效**时，追加注入子模块（仅加载我们的注入 DLL）
- `ModuleData/Languages/`：
  - 生成/规范化 `language_data.xml`
  - 生成目标语言目录（`CNs`/`CNt`）并写入对应语言文件
- `ModuleData/Languages/runtime_localization.json`：
  - 仅当存在“硬编码注入条目且译文有效”时生成

默认输出位置：
- 外置包（external）默认输出到原 Mod 的同级目录（即模块根目录的父目录），可由 CLI `--output` 或 GUI“自定义”覆盖。

## 翻译条目数据模型（用于展示：所属文件 / id / text / 译文）

### Project Record（翻译工程记录）

建议路径：
- 默认：`<工具目录>/data/records/<ModuleId>.mbcns_project.json`

顶层结构（示意）：
- `schemaVersion`：整数
- `moduleId`：从 `SubModule.xml` 的 `<Id value="...">` 获取；缺失则用目录名
- `moduleRootPath`：绝对路径（仅用于本地；不进入打包产物）
- `targetLanguage`：`zh-CN` / `zh-TW` ...
- `scanDll`：bool
- `generatedAtUtc`：时间戳
- `entries[]`：翻译条目

条目结构（示意）：
- `entryKind`：`LanguageString` / `MissingIdFromReferences` / `DllTextObjectIdMissing` / `DllTextObjectHardcoded`
- `sourceFile`：相对路径（XML）或程序集名（DLL）
- `id`：string（`{=id}` 的 id 或 `auto_...`）
- `sourceText`：源文（去掉 `{=id}` 前缀后的默认文本，或硬编码原文）
- `targetText`：译文（可编辑）
- `sourceTextBase64`：`UTF-8(sourceText)` 的 Base64（用于注入匹配与抗转义）
- `contexts[]`：可选（仅 DLL 条目）：`assembly/type/method` 用于审计与定位

### runtime_localization.json（数组结构，注入使用）

建议路径：
- `<packageRoot>/ModuleData/Languages/runtime_localization.json`

结构（示意）：
- `schemaVersion`
- `targetLanguage`
- `generatedAtUtc`
- `gameVersionGate`：版本门禁（见下）
- `entries[]`：
  - `id`
  - `sourceText`
  - `sourceTextBase64`
  - `targetText`
  - `contexts[]`（可选）

## 运行时注入子模块（方案A）

### 核心思路

外挂汉化包内置一个注入 DLL（注入子模块），在运行时对 `TextObject` 构造首参进行“精确匹配替换”：

- 若 `TextObject` 构造入参字符串的 Base64 命中 `runtime_localization.json` 的某条 `sourceTextBase64`，则把入参替换为 `targetText`。
- 若版本门禁不满足或加载失败，则禁用注入（严格安全优先）。

### 兼容性与失败隔离

- 使用 Harmony `PatchAll`。
- 动态枚举 `TextObject` 的所有 `.ctor(string, ...)` 重载进行 patch：
  - 对每个重载 `try/catch`，单个签名失败不影响整体；
  - 注入逻辑内部也 `try/catch`，任何异常都应吞掉并回退原文，避免崩溃。

### 版本门禁（严格安全优先）

注入 DLL 启动时必须完成版本检测：
- 通过反射读取核心程序集（`TaleWorlds.*`）版本，或读取可用的游戏版本对象；
- 若“无法可靠读取版本”或“不在允许范围”，则完全禁用注入（不 patch / 不替换）。

门禁配置来源：
- `runtime_localization.json` 的 `gameVersionGate` 字段（例如允许的核心程序集版本列表/范围）。

## 占位符安全策略

必须保留以下 token 的内容与数量一致：
- `{...}`（包括 `{PLAYER.NAME}`、`{0}`、`{=someId}` 等）
- `{{...}}`
- `<...>`
- `%...%`

策略：
- 翻译前对 token 做保护替换（生成临时占位符），翻译后还原；
- 校验失败则回退源文，不写入译文（避免注入/语言文件写入导致游戏运行期格式崩坏）。

## CLI 形态（破坏性替换）

改为子命令三段式（WPF 与 CLI 行为一致）：

- `scan`：生成/刷新工程记录文件
- `translate`：填充记录中空译文（谷歌翻译），并写入缓存
- `package`：从记录生成外挂汉化包（语言文件 + 可选注入 DLL + runtime_localization.json）

## WPF 形态（主流程按钮）

- “选择 Mod 根目录”
- “扫描解析 DLL（仅 TextObject 字面量）”复选框
- “扫描”按钮：生成记录并展示 DataGrid（四列：所属文件 / id / text / 译文）
- “开始翻译”按钮：填充空译文
- DataGrid 手工校对编辑
- “打包”按钮：生成外挂汉化 Mod 并保存记录

## 测试与验收

必须补齐关键路径测试（目标覆盖率 ≥ 90%，若当前基线不足需在实现计划中提出补齐方案）：

1. DLL 扫描仅命中 `TextObject::.ctor(string, ...)` 字面量
2. `{=id}` 去重：遍历多语言文件、多文件名、优先 `CNs`
3. `runtime_localization.json` 数组结构与 Base64 匹配稳定（含换行符）
4. 占位符安全：破坏 token 的译文必须被拒绝
5. 打包产物：
   - 无硬编码条目时不生成注入模块与 runtime 文件
   - 有硬编码条目时生成注入模块并在 `SubModule.xml` 注册

## 迁移策略

无迁移，直接替换。
